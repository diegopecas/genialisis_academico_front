ARQUITECTURA DE PERMISOS IMPLEMENTADA:

1. TABLAS EN BD (ya creadas en cada tenant):
   - modulos_sistema: árbol recursivo de la app (id, id_padre, nombre, ruta, descripcion TEXT, descripcion_texto TEXT, componente, icono, orden, imagenes JSON, activo)
   - permisos: catálogo de permisos (id, id_modulo, codigo, nombre, descripcion TEXT, activo)
   - permisos_x_rol: asignación many-to-many (id, id_rol, id_permiso)
   - usuarios: tiene campo super_admin TINYINT(1)
   - roles: (id, nombre) — Administrativo(1), Docente(2), Asistente Administrativo(3)
   - roles_x_usuario: (id, id_usuario, id_rol)

2. BACKEND (PHP):
   - jwt.service.php: genera JWT con permisos incluidos en el payload (segundo parámetro $permisos en generarToken). El JWT contiene: id, id_persona, usuario, primer_nombre, primer_apellido, super_admin, permisos[]
   - permisos.service.php: valida permisos DESDE EL JWT sin consultar BD. Métodos: validar($userData, $codigoPermiso) y validarAlguno($userData, $codigosPermisos). Si es super_admin o tiene ['*'], permite todo.
   - usuarios.service.php: en getUsuarioByUsuarioAndClave() obtiene permisos del usuario, los pasa a generarToken() y los incluye en la respuesta JSON
   - Para proteger un endpoint: $userData = JWTService::requerirAutenticacion(); PermisosService::validar($userData, 'codigo.permiso');

3. FRONTEND (Angular):
   - permisos.service.ts: lee permisos de sessionStorage. Métodos: tienePermiso(codigo), tieneAlgunPermiso(codigos[]), tieneTodosLosPermisos(codigos[]), esSuperAdmin()
   - permisos.guard.ts: guard independiente del AuthGuard. Si la ruta tiene data.permiso lo valida, si no lo tiene pasa. Si es super_admin pasa siempre. Redirige a /menu con Swal si no tiene permiso.
   - En app.routes.ts: canActivate: [AuthGuard, PermisosGuard], data: { permiso: 'codigo.permiso' }
   - En menús/submenús HTML: *ngIf="permisosService.tienePermiso('codigo')"
   
   PERMISOS EN COMPONENTES CON TABLA (app-tablas):
   - Se controlan desde el componente HOST, NO desde el TablasComponent
   - En el TS del host: inyectar PermisosService, crear variables (puedeCrear, puedeEditar, puedeEliminar) y configurar en ngOnInit con permisosService.tienePermiso('codigo')
   - Las acciones custom se agregan condicionalmente al array acciones[] según permisos
   - En el HTML del host pasar: [mostrarCrear]="puedeCrear" [accionEditar]="puedeEditar" [accionEliminar]="puedeEliminar"
   - El botón Crear del header se controla con [mostrarCrear]="puedeCrear"
   
   Ejemplo:
   ```typescript
   // En el TS
   configurarPermisos(): void {
       this.puedeCrear = this.permisosService.tienePermiso('admin.productos.crear');
       this.puedeEditar = this.permisosService.tienePermiso('admin.productos.editar');
       this.puedeEliminar = this.permisosService.tienePermiso('admin.productos.eliminar');
       if (this.permisosService.tienePermiso('admin.productos.movimientos')) {
           this.acciones.push({ id: 'movimientos', label: 'Movimientos', icono: '/assets/images/producto.png' });
       }
   }
   ```
   ```html
   <!-- En el HTML -->
   <app-header [mostrarCrear]="puedeCrear" ...></app-header>
   <app-tablas [accionEditar]="puedeEditar" [accionEliminar]="puedeEliminar" [acciones]="acciones" ...></app-tablas>
   ```

   - auth.interceptor.ts: ya envía Bearer token en cada request
   - login.component.ts: guarda token en sessionStorage.setItem('token') y usuario completo (con permisos) en sessionStorage.setItem('usuario')

DOCUMENTACIÓN EN modulos_sistema:

La tabla tiene DOS campos de descripción:
- descripcion (TEXT): versión HTML con formato visual. Usa tags básicos: <b>, <u>, <br>, guiones (-) para listas. Para mostrar en la app.
- descripcion_texto (TEXT): versión texto plano sin HTML. Mismo contenido pero sin tags. Para enviar como contexto a una IA.

La documentación se genera LEYENDO EL CÓDIGO (TS + HTML + PHP), NO describiendo imágenes. Para formularios debe ser campo por campo indicando: nombre del campo, tipo de input, si es obligatorio, de dónde se cargan los datos (tabla, servicio), validaciones.

Campo imagenes (JSON): array de rutas relativas a assets/images/. Ejemplo:
'["documentacion/productos-listado.png", "documentacion/productos-editar.png", "documentacion/productos-movimientos.png"]'

Las imágenes las proporciona Diego. Yo (la IA) le indico el nombre exacto con que debe guardarlas en assets/images/documentacion/. Convención de nombres: nombre-del-modulo-vista.png (ej: productos-listado.png, productos-editar.png, datos-maestros-catalogos.png)
